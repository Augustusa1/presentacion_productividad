Flujograma: De Datos Crudos a Indicador de Productividad

Este diagrama visualiza el viaje completo de los datos. Muestra cómo las tres fuentes de datos principales son procesadas en paralelo y luego consolidadas para generar el indicador final de Kg / Hora-Hombre.

Área de Almacén y Despacho

1. Carga de Datos Crudos

Se cargan y unifican los QVDs de movimientos de inventario (actual e históricos).

↓

2. Enriquecimiento con Maestros

Se cruza la información con el `Maestro_Almacenes` para añadir detalles como la sucursal.

↓

3. Clasificación de Operaciones

Paso CRÍTICO: Se aplica la lógica de `IFs` para asignar un `TIPO_OPERACION` (Venta, Recepción, etc.) a cada transacción.

↓

Resultado: Tabla `Movimientos_Inv`

Una tabla limpia con todos los Kilos movidos, clasificados por operación productiva.

Área de Transporte

1. Carga de Datos de Rutas

Se cargan y unifican los QVDs de rutas de Greenmile (actual e históricos).

↓

2. Filtrado de Rutas

Se aplican filtros para excluir rutas de prueba, días de contingencia e inconsistencias.

↓

3. Cruce con Despacho

Se une con los datos de `Despacho y Transporte` para obtener el `TIPO_VIAJE` y el `DISTRIBUIDOR`.

↓

Resultado: Tabla `Rutas_Paradas`

Una tabla limpia con todas las rutas y paradas válidas para medir la productividad.

Área de Personal

1. Carga de Datos de RRHH

Se carga el archivo TR4 con el registro de horas trabajadas de todo el personal.

↓

2. Identificación de Excepciones

Se aplican las reglas para marcar al personal `REUBICADO` o en `SINDICATO`.

↓

3. Agrupación de Áreas

Se crea el campo `AREA_2` para agrupar al personal en Despacho, Transporte y Staff.

↓

Resultado: Tabla `FINAL_TR4_2`

Una tabla con las horas trabajadas, donde el personal no productivo está claramente identificado.

↓

Consolidación Final

Kilos Movidos

De `Movimientos_Inv`

+

Horas Productivas

De `FINAL_TR4_2` (filtrado)

↓
Indicador de Productividad Final

Kg Movidos / Horas-Hombre

¿Cómo sabe el sistema qué es cada movimiento?

El paso más crítico es asignar cada transacción a un **Tipo de Operación**. Esto permite medir únicamente los movimientos que generan productividad. Seleccione un tipo de operación para ver la lógica de Qlik que se aplica.

¿Por qué cada sucursal tiene reglas diferentes?

Una de las mayores complejidades del script es la identificación de transporte tercerizado, ya que la forma de registrarlo no es la misma en todas partes. Explore las particularidades de cada sucursal.

¿Las horas de quiénes se excluyen del cálculo?

Para calcular la productividad real, necesitamos saber qué horas de trabajo cuentan y cuáles no. El script usa "listas especiales" para excluir las horas de personas que no estaban haciendo trabajo operativo.

Análisis de Almacenes y Transferencias

El maestro de almacenes nos permite entender el rol de cada ubicación. Aquí podemos ver cómo interactúan y qué transacciones son vitales para la operación entre ellos.

Anatomía de las Transferencias y Movimientos Clave

Transferencia entre Organizaciones

Mueve producto entre dos "organizaciones" o centros de distribución principales. Es una operación logística clave para balancear inventario a nivel nacional.

CD SCZ (C43) ➔ CD LPZ (C42)
TIPO_TRANS: `Direct Org Transfer`, `Transferencia Org 2 Pasos`
Transferencia Interna de CD

Mueve producto DENTRO del mismo CD. No sale del edificio, solo cambia de área (ej: de recepción a picking).

Dentro de CD SCZ (C43): Área '07' ➔ Área '01'
TIPO_TRANS: `Subinventory Transfer`, `Move Order Transfer`
Abastecimiento a Almacenes Secundarios

Es la salida de un CD principal para abastecer a otro almacén de la red, como una agencia comercial o un CD regional.

CD SCZ (C43) ➔ Agencia Grigota (A01)
TIPO_TRANS: `Int Order Intr Ship`
Recepción de Mercadería Externa

Ingreso de producto desde un proveedor externo (una compra) o la devolución de un cliente.

Proveedor ➔ CD CBBA (C31)
TIPO_TRANS: `PO Receipt`, `RMA Receipt`

Conclusión y Desafíos Clave

El script actual es funcional pero presenta desafíos importantes para su mantenimiento y escalabilidad. La precisión del indicador de productividad depende directamente de la actualización constante de estas reglas.

Alta Dependencia de Valores Manuales

El código contiene listas de IDs de empleados, nombres de conductores y placas de vehículos "quemados" (hardcoded). Cualquier cambio en el personal o la flota requiere una modificación directa del script.

Mantenimiento Constante

Las fechas de contingencia y las condiciones especiales del personal deben ser añadidas manualmente. Esto representa un riesgo de que el cálculo se vuelva obsoleto o impreciso si no se actualiza a tiempo.

Una posible mejora a futuro sería externalizar estas reglas a archivos maestros o tablas de configuración, facilitando su actualización sin necesidad de editar el código fuente del script.
